## Codebase Patterns
- Scraper usa requests.Session com headers HEADERS definidos no topo do módulo
- Parsing HTML usa BeautifulSoup com seletor div.resultadoItem para cada card
- Payload do POST (fetch_page) é idêntico entre tribunais — só mudam a URL (base eproc) e os headers Origin/Referer
- Campos vêm em pares .resLabel / .resValue; no TJSC ficam em div.row, no TJSP/TJRJ vários ficam em <div class="col-12 col-md-4"> — por isso o parser deve iterar .resLabel direto (não só div.row)
- extract_processo usa a.numero-processo como prioridade, fallback para campo PROCESSO (split()[0]) — já limpa lixo tipo "ProCom - Procedimento" do TJSP
- Relator: label varia — RELATOR/RELATORA (TJSC, TJRJ) e MAGISTRADA/MAGISTRADO (TJSP)
- Variáveis de ambiente GCP: GCP_PROJECT_ID, BQ_DATASET_ID, BQ_TABLE_ID (=processos_raw nesta fase, dados BRUTOS sem embedding)
- ID de registro: SHA-256 de "{processo}|{data_julgamento}" — número CNJ já discrimina o tribunal (.8.26.=SP, .8.24.=SC, .8.19.=RJ)
- BigQuery insert: chunks de 256 com retry 3x; mock em teste: patch("screping.bq_writer.bigquery.Client")
- checkpoint.json fica na raiz do repo (não dentro de screping/)
- Usar uv para venv/deps: `uv venv .venv && uv pip install -r requirements.txt --python .venv/bin/python` (não há pip/pip3 no sistema)
- scrape_month aceita failed_pages: list | None = None para coletar falhas sem alterar tipo de retorno
- Padrão de teste de scrape_month: patch("screping.scraper.fetch_page") + patch("time.sleep"); sleep=0 evita lentidão
- 3 páginas consecutivas sem resultado (ou com falha) encerram o mês
- Tribunais drop-in confirmados na Fase 0: TJSC, TJSP (eproc1g.tjsp.jus.br), TJRJ (eproc1g.tjrj.jus.br). TJRS/TJMG/etc usam outra implementação eproc (lançam exceção)
- Embedding de produção (fora desta fase): intfloat/multilingual-e5-base, prefixo "passage: {text}"
